# CONTENT_TONE.md

## Portfolio Voice

The portfolio voice should feel:

- warm
- youthful
- creative
- tidy
- personal
- friendly
- recruiter-friendly
- playful but still professional

The writing should sound like a real junior developer who enjoys building, organizing, cleaning data, and making interfaces feel friendly.

Do not make the copy sound stiff, corporate, robotic, or overhyped.

## Overall Style

Use clear professional structure first, then add small playful details.

Good balance:

```text
Built a full-stack e-commerce website for a family Muslim fashion store from Tanah Abang, covering product browsing, checkout, shipping, payment, and admin stock management.
Then a playful line is okay:

A lot of tiny details, one much tidier store workflow.

Avoid
Avoid phrases that sound exaggerated:

best-in-class
highly advanced

Avoid making small projects sound bigger than they are.
Avoid too many jokes inside technical bullets.

Section Tone Guide
Hero
Tone: warm, confident, personal.
Good direction:

A Computer Science student who enjoys turning messy data into readable insights and building tidy web experiences that feel friendly to use.
About
Tone: personal and reflective.
Mention:

Computer Science student
data analysis
software development
Python, SQL, C++, JavaScript, TypeScript, and Java
project coordination
tidy systems
Keep it human, not CV-like.

Skills
Tone: honest and clear.
Do not say “expert in everything.”
Use skill chips and friendly labels.
Programming language order:

Python
SQL
C++
JavaScript
TypeScript
HTML
CSS
Java
Projects
Tone: clear, evidence-based, lightly playful.
Each project should clearly show:

project name
year
role
category
tech stack
short description
my part
demo link if available
GitHub link if available
screenshot/gallery if available
Good project copy pattern:

[Project Name] is a [type of project] built to [main purpose]. My part focused on [contribution]. The goal was to make [workflow/problem] feel [clearer/easier/tidier].
Experience
Tone: professional first, warm second.
Use role, company/project, period, and contribution clearly.
Tiny human detail is allowed, but bullets should stay achievement-focused.
Good direction:

Worked on an internal sales funnel dashboard across frontend, backend, authentication, database design, upload flow, and ETL support.
Certificates
Tone: “proof shelf,” not “trophy wall.”
Certificates should support credibility without feeling too loud.
Good phrases:

tiny proof shelf
learning receipts
small milestones
structured learning
little receipts of learning
Contact
Tone: friendly and open.
Good direction:

Open to internships, freelance projects, collaboration, or a friendly chat about data, dashboards, and web apps.
Language Rule
Main website language: English.
Use natural English.
Do not mix Indonesian and English on the live website unless it is a project name or issuer name.

Copywriting Rule
Every card should answer:

What is it?
What did Abdul do?
What tools were used?
Why does it matter?
Final Quality Bar
The copy should feel like:

friendly enough to be memorable,
professional enough to send to HR,
specific enough to prove the work is real,
and soft enough to match the approved design.
